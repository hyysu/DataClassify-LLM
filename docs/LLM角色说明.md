# LLM 在数据分类分级工具中的角色说明

## 1. 结论

LLM 在本项目中不是必需组件，而是增强组件。

不接入 LLM，项目仍然可以完成：

```text
字段输入 -> 字段分类打标 -> 本地规则分级 -> 输出报告
```

接入 LLM 后，项目可以增强：

```text
复杂场景理解 -> 分级理由生成 -> 不确定字段复核 -> 合规建议生成
```

因此，本项目的合理设计是：

```text
贝叶斯/规则负责稳定识别
LLM 负责复杂推理和解释增强
人工负责最终确认
```

## 2. LLM 在项目中的角色

### 2.1 分级辅助判断器

贝叶斯模型更适合回答：

```text
这个字段是什么类型？
```

例如：

- `mobile_phone` -> 手机号
- `id_card_no` -> 身份证件号
- `contract_amount` -> 合同金额
- `source_code_path` -> 源代码路径

LLM 更适合回答：

```text
结合字段类型、业务场景、样例摘要和规则，这个字段应该定为几级？
```

例如：

同样是 `amount`：

- 如果是订单金额，可能是 L3；
- 如果是公开统计金额，可能是 L1/L2；
- 如果是未披露财务报表金额，可能是 L4。

这种场景判断是 LLM 的优势。

### 2.2 解释生成器

分类分级工具不能只输出：

```text
level = L4
```

还应该说明：

```text
为什么是 L4？
命中了什么规则？
可能有什么风险？
应该采取什么保护措施？
```

LLM 可以把模型和规则结果转成更像专家报告的语言。

### 2.3 疑难字段复核员

当字段名、注释、样例都很模糊时，传统模型容易不确定。

例如：

- `value`
- `content`
- `detail`
- `extra_info`
- `biz_data`

这些字段需要结合：

- 表名
- 业务场景
- 上下文字段
- 数据样例摘要
- 企业内部规则

LLM 可以作为“二次判断器”，只处理置信度低或高风险字段。

### 2.4 合规建议生成器

LLM 可以根据分级结果生成保护建议，例如：

- 是否需要脱敏展示
- 是否需要加密存储
- 是否需要严格访问控制
- 是否需要操作审计
- 是否需要人工复核
- 是否可能涉及个人信息、敏感个人信息、重要数据

这些建议可以放到报告中，提升工具的咨询交付价值。

## 3. 为什么不加 LLM 也能实现

因为本项目的基础任务可以拆成两个确定性问题：

### 3.1 字段分类可以用传统机器学习完成

字段分类本质上是监督学习问题：

```text
字段名 + 字段注释 + 数据类型 + 样例摘要 + 业务场景 -> 分类标签
```

当前项目已经使用：

- 贝叶斯分类器
- 字段关键词
- 正则模式
- 标签体系

这些足以完成大部分字段打标。

例如：

- `phone_number`、`手机号` -> 手机号
- `id_card_no`、`身份证号码` -> 身份证件号
- `bank_card_no`、`银行卡号` -> 银行卡号
- `source_code_path`、`源代码路径` -> 商业秘密

这些判断不需要 LLM。

### 3.2 数据分级可以先用规则完成

很多分级判断可以规则化：

```text
身份证件号 -> L4
银行卡号 -> L4
密码/密钥/Token -> L4
手机号/地址/设备ID -> L3
公开公告 -> L1
一般内部流程状态 -> L2
```

当前项目已经通过 `RuleBasedGrader` 实现了这种规则分级。

这类规则判断具有几个优点：

- 稳定
- 可解释
- 可复现
- 成本低
- 不依赖网络
- 不会把数据发给外部服务

因此，不接 LLM，工具仍然能跑通。

## 4. LLM 不适合作为唯一核心的原因

### 4.1 结果不够稳定

LLM 同一个输入在不同时间、不同模型版本、不同提示词下，可能输出不同结果。

数据分类分级属于合规和安全场景，结果需要可复核、可追溯。

### 4.2 存在数据泄露风险

如果直接把真实字段样例、个人信息、合同数据、财务数据传给外部大模型 API，可能产生新的数据安全风险。

因此应尽量只传：

- 字段名
- 字段注释
- 数据类型
- 脱敏样例
- 统计摘要
- 本地模型初判结果

### 4.3 成本和性能不可控

如果每个字段都调用一次 LLM：

- 成本会上升
- 响应会变慢
- 大批量表结构扫描效率较低

更合理的方式是：

```text
高置信度字段：本地模型 + 规则直接输出
低置信度字段：调用 LLM 辅助判断
高风险字段：调用 LLM 生成理由和保护建议
```

### 4.4 难以直接承担责任

分类分级结果最终要能给出依据。

LLM 的回答可以作为参考，但最终仍应落到：

- 标签体系
- 分级规则
- 法规标准
- 企业制度
- 人工复核记录

## 5. 推荐架构

推荐架构如下：

```text
CSV/数据库字段
    |
    v
字段特征提取
    |
    v
贝叶斯分类器 + 关键词/正则纠偏
    |
    v
本地规则分级
    |
    +---- 高置信度 -> 直接输出报告
    |
    +---- 低置信度/高风险 -> LLM 辅助判断
                                |
                                v
                             人工复核
                                |
                                v
                             结果回流样本集
```

## 7. 后续接入 LLM 的建议

建议不要一开始让 LLM 处理所有字段。

更推荐：

1. 只对 `requires_review = True` 的字段调用 LLM。
2. 只对 L3/L4 字段调用 LLM 生成详细理由。
3. 只传脱敏样例或摘要，不传真实敏感数据。
4. LLM 输出必须是 JSON。
5. LLM 结果必须可人工修改。
6. 人工修改结果要回流训练样本。

这样可以兼顾：

- 成本
- 安全
- 准确性
- 可解释性
- 可复核性
