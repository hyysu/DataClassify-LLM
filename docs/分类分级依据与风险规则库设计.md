# 分类分级依据与风险规则库设计

## 1. 设计目标

本设计用于支撑当前“数据分类分级工具”的后续完善，重点解决三个问题：

1. 分类标签体系不能只停留在 demo，需要能覆盖个人信息、重要数据、企业经营、AI 安全、供应链和碳排放等场景。
2. 分级规则不能只依赖大模型自由判断，需要有法律法规、国家标准和行业规则依据。
3. 风险数据集和规则库需要结构化，后续可以直接接入贝叶斯分类、规则分级和 LLM 分级模块。

## 2. 主要依据

### 2.1 基础法律法规

- 《中华人民共和国数据安全法》
  - 支撑点：国家建立数据分类分级保护制度；对重要数据加强保护；开展数据安全风险监测、风险评估、事件处置。
  - 对项目的作用：作为“分类分级”和“重要数据候选识别”的上位依据。

- 《中华人民共和国个人信息保护法》
  - 支撑点：区分个人信息和敏感个人信息；敏感个人信息包括生物识别、宗教信仰、特定身份、医疗健康、金融账户、行踪轨迹以及未满十四周岁未成年人个人信息等。
  - 对项目的作用：作为个人信息分类标签、敏感个人信息识别和高等级保护规则的依据。

- 《中华人民共和国网络安全法》
  - 支撑点：网络运营者安全保护义务、个人信息保护义务、网络安全事件处置、关键信息基础设施保护。
  - 对项目的作用：作为系统日志、账号凭据、网络安全事件、关键信息基础设施相关数据分级的依据。

- 《网络数据安全管理条例》
  - 支撑点：围绕网络数据处理活动，强化个人信息、重要数据、网络数据安全保护和风险处置要求。
  - 对项目的作用：作为网络数据处理场景下风险识别、重要数据处理和事件处置规则的依据。

### 2.2 国家标准和推荐性规范

- GB/T 43697-2024《数据安全技术 数据分类分级规则》
  - 支撑点：提供数据分类分级的一般方法，强调结合数据所属领域、业务属性、数据特征、影响对象和影响程度开展分级。
  - 对项目的作用：作为本项目“分级矩阵”的主要标准依据。

- GB/T 35273-2020《信息安全技术 个人信息安全规范》
  - 支撑点：列举个人信息和敏感个人信息示例，覆盖身份、联系方式、账户、财产、健康、生物识别、行踪轨迹等。
  - 对项目的作用：作为个人信息字段分类标签和正则/语义特征设计的依据。

- GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》
  - 支撑点：提出不同安全保护等级下的安全控制要求。
  - 对项目的作用：作为输出“保护建议”的参考依据。

### 2.3 人工智能安全相关文件

- 《生成式人工智能服务管理暂行办法》
  - 支撑点：生成式人工智能服务应坚持发展和安全并重；训练数据应具有合法来源；涉及个人信息应符合法定条件；应提升训练数据真实性、准确性、客观性、多样性；发现违法内容和违法使用应采取处置措施并报告。
  - 对项目的作用：作为 AI 安全风险类型、训练数据风险、输出内容风险和风险报送字段设计的依据。

- 《互联网信息服务算法推荐管理规定》
  - 支撑点：算法推荐服务应履行安全管理、用户权益保护和备案等义务。
  - 对项目的作用：作为算法推荐风险、模型服务风险和用户权益风险的依据。

- 《互联网信息服务深度合成管理规定》
  - 支撑点：深度合成服务应进行标识、履行安全管理义务，防止违法和不良信息传播。
  - 对项目的作用：作为深度合成、伪造内容、标识缺失等 AI 风险标签的依据。

### 2.4 行业场景依据

- 《工业和信息化领域数据安全管理办法（试行）》
  - 支撑点：工业和信息化领域数据可分为一般数据、重要数据、核心数据。
  - 对项目的作用：作为供应链、工业生产、关键零部件、工业运行数据分级的依据。

- 《碳排放权交易管理暂行条例》
  - 支撑点：重点排放单位需要依法开展碳排放数据核算、报告、核查和配额清缴。
  - 对项目的作用：作为碳排放数据、碳核算数据、配额交易数据、核查报告数据风险分类的依据。

## 3. 分类标签体系设计

建议采用“两级标签 + 场景标签 + 风险标记”的结构。

### 3.1 一级分类

| 一级分类 | 说明 |
|---|---|
| 个人信息 | 能够识别或关联自然人的信息 |
| 敏感个人信息 | 泄露或非法使用后容易导致人格尊严受侵害或人身、财产安全受危害的信息 |
| 组织经营信息 | 企业经营、管理、客户、供应商、合同、价格等信息 |
| 业务交易信息 | 订单、支付、交易、物流、结算等业务流转信息 |
| 系统运行与安全数据 | 日志、账号、密钥、IP、设备、漏洞、告警等 |
| 人工智能安全数据 | 训练数据、标注数据、模型输入输出、评测数据、风险事件数据 |
| 供应链数据 | 供应商、采购、库存、物流、关键部件、供应中断等 |
| 碳排放与环境数据 | 排放量、能耗、碳配额、碳交易、核查报告、生产工艺能耗等 |
| 公共或普通业务数据 | 已公开、低敏、对权益影响较低的数据 |
| 其他 | 无法明确归类或需要人工复核的数据 |

### 3.2 二级标签建议

| 一级分类 | 二级标签 |
|---|---|
| 个人信息 | 个人基本信息、个人联系方式、个人网络身份标识、个人教育职业信息 |
| 敏感个人信息 | 个人身份信息、个人生物识别信息、个人财产信息、个人健康生理信息、个人行踪轨迹、未成年人个人信息 |
| 组织经营信息 | 客户信息、供应商信息、合同信息、报价信息、经营分析信息、商业秘密候选 |
| 业务交易信息 | 订单信息、支付信息、发票信息、物流信息、售后信息、交易流水 |
| 系统运行与安全数据 | 系统日志、访问日志、账号权限、认证凭据、密钥令牌、漏洞告警、设备指纹 |
| 人工智能安全数据 | 训练数据、标注数据、提示词数据、模型输出、评测样本、风险事件记录 |
| 供应链数据 | 供应商名录、采购价格、库存数据、物流路径、关键零部件、供应中断风险、供应商信用 |
| 碳排放与环境数据 | 企业排放量、能源消耗、碳核算参数、碳配额、碳交易、核查报告、设备能耗、工艺排放因子 |

### 3.3 风险标记

标签之外，再增加风险标记，避免分类标签过度膨胀：

| 风险标记 | 含义 |
|---|---|
| personal_info | 个人信息 |
| sensitive_personal_info | 敏感个人信息 |
| important_data_candidate | 重要数据候选 |
| core_data_candidate | 核心数据候选，仅作预警，不自动定性 |
| trade_secret_candidate | 商业秘密候选 |
| credential_or_secret | 密钥、令牌、口令等安全凭据 |
| ai_safety_risk | AI 安全风险 |
| supply_chain_risk | 供应链安全风险 |
| carbon_data_risk | 碳排放数据风险 |
| cross_border_sensitive | 涉及数据出境敏感性 |
| need_manual_review | 需要人工复核 |

## 4. 分级规则设计

当前工具可以继续使用 L1-L4，但建议增加“核心数据候选”标记，不在 demo 中直接输出 L5。

### 4.1 等级定义

| 等级 | 名称 | 判断原则 |
|---|---|---|
| L1 | 公开或低敏数据 | 已公开或泄露后影响较小 |
| L2 | 内部一般数据 | 内部业务数据，泄露后造成一般管理影响 |
| L3 | 敏感数据 | 涉及个人信息、经营敏感、交易数据、系统运行数据，泄露后可能损害个人或组织权益 |
| L4 | 高敏感或重要数据 | 涉及敏感个人信息、认证凭据、重要数据候选、关键供应链、重大经营或安全风险 |

### 4.2 分级判断维度

| 维度 | 说明 |
|---|---|
| 影响对象 | 国家安全、公共利益、个人权益、组织权益、市场秩序、生态环境治理 |
| 影响程度 | 一般影响、严重影响、特别严重影响 |
| 数据属性 | 个人信息、敏感个人信息、重要数据候选、商业秘密候选、系统安全凭据 |
| 数据规模 | 单条、批量、大规模、长期连续积累 |
| 数据精度 | 模糊、一般、精确、可定位到个人/企业/设施 |
| 数据场景 | 普通业务、关键业务、关键信息基础设施、公共服务、监管报送 |
| 聚合风险 | 多字段组合后是否可识别个人、还原交易、推断商业秘密 |
| 出境风险 | 是否存在跨境提供、境外访问、境外模型调用 |
| 时效性 | 是否涉及实时运行、实时定位、实时交易、实时供应链状态 |

### 4.3 基础规则

| 规则 | 建议等级 |
|---|---|
| 已公开公告、公开新闻、公开产品说明 | L1 |
| 普通内部字段、一般统计字段、普通状态码 | L2 |
| 个人基本信息、联系方式、网络身份标识 | L3 |
| 订单、交易、合同、客户、供应商、经营分析数据 | L3 |
| 系统日志、访问日志、IP、设备信息 | L3 |
| 身份证件、生物识别、金融账户、健康医疗、精准定位、未成年人个人信息 | L4 |
| 密码、密钥、token、client_secret、证书私钥 | L4 |
| 重要数据候选、关键供应链、关键工业运行、重大碳排放核算数据 | L4 |
| 分类置信度低、规则冲突、LLM 和规则不一致 | 不自动降级，标记 need_manual_review |

### 4.4 LLM 分级约束

LLM 不应直接自由判断等级，应按以下顺序工作：

1. 接收贝叶斯分类结果、置信度、证据特征、样本画像。
2. 读取规则库中命中的规则。
3. 在规则候选等级基础上补充理由。
4. 不允许把强规则命中的 L4 降为 L2。
5. 输出必须包含：等级、理由、命中依据、保护建议、是否需要人工复核。

## 5. 风险数据集设计

风险数据集不是单纯训练集，而是“分类训练 + 分级验证 + 风险研究 + 案例说明”的统一数据资产。

### 5.1 字段结构

| 字段 | 说明 |
|---|---|
| record_id | 风险样本编号 |
| domain | 场景领域：personal、ai、supply_chain、carbon、system、business |
| table_name | 表名或数据对象名称 |
| table_comment | 表说明 |
| column_name | 字段名 |
| column_comment | 字段说明 |
| data_type | 数据类型 |
| sample_profile | 样本画像，不保存真实敏感值 |
| regex_features | 正则特征，如手机号、身份证、IP、URL、金额、日期 |
| category_label | 分类标签 |
| risk_tags | 风险标记 |
| suggested_level | 建议等级 |
| harm_object | 影响对象 |
| harm_degree | 影响程度 |
| risk_scenario | 典型风险场景 |
| legal_basis | 法规或标准依据 |
| rule_ids | 命中的规则编号 |
| controls | 保护建议 |
| need_report | 是否可能需要报告 |
| report_fields | 如发生风险，建议报送的信息字段 |
| review_notes | 人工复核说明 |

## 8. 后续落地路线

### 阶段一：规则表落地

1. 建立 `data/risk_rule_catalog.csv`。
2. 建立 `data/risk_dataset_template.xlsx`。
3. 先录入个人信息、系统安全、供应链、碳排放、AI 风险五类规则。
4. 在分级模块中增加规则优先级，避免 LLM 把强规则命中的高风险字段降级。

### 阶段二：风险数据集建设

1. 生成一批合成风险字段样本。
2. 人工补充真实业务风格字段，但不保存真实敏感值。
3. 每条样本标注分类标签、风险标签、建议等级、依据文件、规则编号。
4. 将风险数据集拆成训练集、验证集、最终测试集。

### 阶段三：LLM 分级增强

1. LLM 输入中加入命中规则。
2. LLM 输出中强制包含依据和命中规则编号。
3. 对 L4 强规则设置不可降级约束。
4. 对规则冲突、置信度低、模型输出不稳定的结果标记人工复核。

### 阶段四：专题研究沉淀

1. 人工智能安全风险研究：形成风险类型、事件报送字段、处置流程。
2. 供应链数据安全研究：形成供应链数据目录、风险场景、重要数据候选规则。
3. 碳排放数据安全研究：形成碳排放数据目录、核算/交易/核查风险规则。
